## 1. 開発ワークフロー（必須）

**いきなり実装しない。必ず以下の順序で進める。**

### Step 1: 仕様確認
- 何を作るか、何が完了条件かをユーザーと言語化する
- 曖昧な点は実装前に質問して解消する
- スコープが MVP 範囲に収まっているか確認する

### Step 2: 実装計画
- ファイル単位で変更箇所を列挙する
- 変更の影響範囲（副作用）を明示する
- 計画をユーザーに提示し、承認を得てから実装に入る

### Step 3: 小さく実装
- 計画のステップを 1 つずつ実装する
- 1 ステップ = 1 つの責務（コンポーネント、関数、スキーマ変更など）
- 途中で仕様変更が発生したら実装を止め、Step 1 に戻る

### Step 4: レビュー
- 実装完了後、変更ファイルの一覧を必ず出力する
- 意図した変更以外が混入していないか確認する
- 動作確認の手順を提示する

### 変更ファイル一覧の出力（毎回必須）

実装が完了するたびに以下のフォーマットで出力すること：

```
## 変更ファイル一覧
- src/components/FooBar.tsx  — 新規作成
- src/screens/HomeScreen.tsx — 修正（カメラ起動ロジック追加）
- app.json                   — 修正（パーミッション追加）
```

## 2. MVP スコープ管理

### やること（初期 MVP）
- カメラで数字を読み取り（OCR）
- 手入力レートで円換算
- 換算結果の保存（旅行 1 件、基本的な品目管理）
- 換算モード選択（例：USD→JPY）を事前設定方式で実装

### やらないこと（初期 MVP 外）
- 通貨の自動判定
- 自動レート取得（Pro 版機能）
- 複数旅行・予算管理（Pro 版機能）
- サーバーサイド処理（Turso / Supabase は使わない）
- ソーシャル機能・共有機能

### スコープが広がりそうになったら
「これは MVP に必要か？」と必ず問い返す。
必要でなければ TODO に積んで後回しにする。実装しない。

## 3. 技術スタック

| 用途 | 技術 | 備考 |
|------|------|------|
| アプリ基盤 | Expo + React Native | Expo Go / EAS Build を使用 |
| 言語 | TypeScript | strict モード推奨 |
| ローカル DB | expo-sqlite | サーバーレス。初期はこれのみ |
| 課金 | RevenueCat | 無料版 / Pro 版のサブスク管理 |
| カメラ / OCR | expo-camera + (ML Kit or Vision) | 数字読み取り特化 |
| 状態管理 | 未定（Zustand or Context API） | 大きなライブラリは避ける |

### 使わない技術（初期）
- Turso（リモート SQLite）
- Supabase（BaaS）
- Firebase
- Redux

---

## 4. アプリ仕様メモ

### 無料版
- 手入力レートのみ（自動取得なし）
- 旅行 1 件まで
- 基本的な品目保存

### Pro 版（将来）
- 自動レート取得
- 旅行件数無制限
- 予算管理機能
- 価格：未定

### OCR 仕様
- 数字・記号（`.`, `,`）の読み取りに特化
- 多言語対応は不要（数字は共通）
- 読み取り精度が低い場合は手入力で補正できる UI を用意

### 換算モード
- 通貨自動判定なし（初期）
- ユーザーが事前に「USD→JPY」などのモードを選択
- 設定した換算モードはローカル DB に保存
